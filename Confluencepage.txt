Summary

Fix access removal flow to allow re-request of access for impacted users

Description

Currently, when a user is removed from the RAP application, the access removal logic updates the EZAC2 flag to 0 in the relevant tables. However, for certain users, existing records in the METRIC_ACCESS table prevent them from raising access requests again.

This behavior is causing access re-provisioning issues for a limited set of users, even though they are no longer active in the application.

A revised approach has been tested successfully in the unit environment, where access removal is handled in a way that ensures:

Existing access records do not block future access requests

Access re-provisioning works as expected

This ticket tracks the production implementation of the validated solution for impacted users only.

Impact

Affects only specific users facing access re-request issues

No impact to existing valid users

No application downtime

Environment

Production (implementation via approved Change Request)

Acceptance Criteria

Impacted users can successfully raise access requests after removal

No unintended access is retained

No regression to existing access flows
