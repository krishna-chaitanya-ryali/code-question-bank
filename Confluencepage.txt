Metric Auto Cascade Enhancement
Overview
The Metric Auto Cascade Enhancement automates the cascading of metrics across the RMM hierarchy. The enhancement supports both Metric Cascade and Metric Removal Cascade, ensuring that metric additions, updates, and removals propagate correctly through the hierarchy while maintaining complete audit history, versioning, and lifecycle management.
Business Objective
The enhancement provides:
Automatic cascading of metrics across child RMMs.
Version-based cascade tracking.
Support for re-cascade scenarios.
Support for metric removal propagation.
Lifecycle management across multiple hierarchy levels.
Complete audit trail for every cascade/remove action.
Dashboard synchronization across all hierarchy levels.

Important Design Principles
1. Never update existing Metric Detail IDs.
Always create a new Metric Detail ID when adding to a dashboard.
2. Never physically delete Cascade history.
Only update lifecycle/status.
3. Every Cascade must create an MCA record.
No exception.
4. Every Remove Cascade must create an MCA Remove record.
No exception.
5. MCL stores hierarchy.
MCA stores activity history.
6. Versioning is mandatory.
Never reuse versions.
7. Parent and Child dashboards have different Metric Detail IDs.
Never assume Metric Detail IDs remain the same across dashboards.
8. Remove requests move level-by-level.
A child cannot receive a Remove request until its parent has processed the previous level.

Lessons Learned
Keep hierarchy (METRIC_CASCADE_LOCATIONS) and audit history (METRIC_CASCADE_ACTIONS) separate.
Metric Detail IDs are dashboard-specific and should never be shared across hierarchy levels.
Versioning simplifies re-cascade and removal tracking.
Ownership (VISIBLE_AT_RMM_ID) and pending work (REMOVE_PENDING_FOR_RMM_ID) should be treated as separate concepts.
Preserve complete history rather than deleting records; lifecycle flags and audit tables make troubleshooting and reporting much easier.
