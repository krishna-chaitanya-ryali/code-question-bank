**Subject:** Discussion on RAP Enhancement – Access Request Form & Location Menu Visibility

Hi Team,

I would like to schedule a quick discussion to review the enhancement request related to the RAP **Access Request Form and Location Menu**.

**Background**
Currently, the RAP application uses the **ACTIVE** flag in the RAP_RMM_GROUP table to control visibility of locations. If ACTIVE = 'Y', the location is visible in both:

* RAP menu
* Access request form dropdown

If ACTIVE = 'N', the location is hidden from both.

However, the new requirement introduces **standalone/hidden locations** which should:

* Not appear in the RAP menu
* Still be available in the Access Request form for users to request access.

**Proposed Solution**
To support this requirement, we are proposing to separate the visibility logic by introducing two columns in RAP_RMM_GROUP:

* SHOW_IN_MENU
* SHOW_IN_ACCESS_REQ

This will allow us to control menu visibility and access request visibility independently.

Example behaviour:

Normal location
SHOW_IN_MENU = Y
SHOW_IN_ACCESS_REQ = Y

Standalone location
SHOW_IN_MENU = N
SHOW_IN_ACCESS_REQ = Y

Inactive location
SHOW_IN_MENU = N
SHOW_IN_ACCESS_REQ = N

**Discussion Points**

1. Validate the proposed database design change.
2. Confirm impact on downstream systems (QuickSense ETL).
3. Confirm how standalone locations should behave in reporting/data pipelines.
4. Agree on implementation approach and rollout plan.

Please share your thoughts during the discussion.

Thanks.

1. Current System Behaviour (Based on your explanation)

Table: RAP_RMM_GROUP

Important column:

ACTIVE (Y / N)

Current logic:

ACTIVE	Menu	Access Request Form
Y	Visible	Visible
N	Hidden	Hidden

So one column is controlling two behaviours:

Menu visibility

Access request visibility

2. New Requirement

Now there is a new category called Standalone / Hidden RMM.

These should behave like this:

Type	Menu	Access Request
Normal Active	Visible	Visible
Standalone RMM	Hidden	Visible
Inactive	Hidden	Hidden

So menu visibility and access request visibility must be controlled separately.

3. Your Proposed Solution

You suggested adding a column:

STANDALONE_RMM (Y/N)

And then modifying queries.

Your idea is correct, but there is a better design approach.

4. Recommended Design (Best Practice)

Instead of STANDALONE_RMM, create two explicit columns.

Option (Best Approach)

Add:

SHOW_IN_MENU        CHAR(1)
SHOW_IN_ACCESS_REQ  CHAR(1)

Example table:

RMM	ACTIVE	SHOW_IN_MENU	SHOW_IN_ACCESS_REQ
Global CMB	Y	Y	Y
Global GBM	Y	Y	Y
Standalone APP	Y	N	Y
Deprecated APP	N	N	N
5. Advantages of this Design
1️⃣ More Flexible

Future requirements become easy.

Example:

Scenario	Possible
Hidden everywhere	YES
Only menu	YES
Only access form	YES
Both	YES

Your STANDALONE_RMM column cannot support all combinations.

2️⃣ Cleaner Queries

Menu query:

SELECT *
FROM RAP_RMM_GROUP
WHERE ACTIVE = 'Y'
AND SHOW_IN_MENU = 'Y';

Access request query:

SELECT *
FROM RAP_RMM_GROUP
WHERE ACTIVE = 'Y'
AND SHOW_IN_ACCESS_REQ = 'Y';
3️⃣ No Logic Confusion

Right now ACTIVE is overloaded.

It is controlling:

existence

menu

access form

Separating responsibilities is clean architecture.

6. If DB Change Is Not Allowed (Alternative)

If they don't allow schema changes, then your approach works:

Add column:

STANDALONE_RMM CHAR(1)

Menu query:

WHERE ACTIVE = 'Y'
AND NVL(STANDALONE_RMM,'N') = 'N'

Access request query:

WHERE ACTIVE = 'Y'

But this is less flexible long-term.

7. What I Would Recommend in Production

As a 10-year Python/backend developer like you, I strongly recommend:

SHOW_IN_MENU
SHOW_IN_ACCESS_REQ

because enterprise systems eventually grow and new combinations will appear.

8. Migration Plan

Step 1 — Add columns

Step 2 — Update existing data
