Servers Included in CR
You can say the change request covers the following environments for safety and rollback readiness:
Application Server – PROD
Application Server – DR
Database Server – PROD
Database Server – DR
Even though the primary execution is on PROD, DR servers are included for consistency and fallback readiness.
Reason for Including All 4 Servers
You can explain like this during the CAP call:
1. Production Application Server
Deployment of application code changes related to User Recertification logic
Scheduler execution for:
Email trigger
System revoke job
Configuration updates if required
2. Production Database Server
Execution of DB schema changes
Table updates for:
RECERTIFICATION_EMAIL_LIST
USER_TAB updates for Line Manager data
Validation queries and data verification
3. DR Application Server
Included for environment consistency
Ensures DR server remains aligned with production codebase
Avoids failover mismatch issues
4. DR Database Server
Included to maintain schema synchronization
If DR failover occurs, recertification process continues without issues
Implementation Plan (CAP Call Version)
If they ask, you can say:
Take database backup before starting the activity.
Deploy application code changes through Jenkins pipeline.
Execute required DB scripts for schema/data updates.
Populate Line Manager data in USER_TAB.
Verify application logs and scheduler execution.
Validate recertification functionality in production.
Verification Steps
After deployment:
Verify DB changes applied successfully.
Confirm Line Manager data populated correctly.
Validate scheduler logs via Splunk.
Confirm application health and access functionality.
Perform sample recertification test with limited users.
Backout Plan
If any issue occurs:
Stop scheduler jobs related to recertification.
Rollback application deployment using previous stable build.
Restore database from backup if required.
Validate application functionality post rollback.
